# SKILL.md Template for Azure SDK for Python Package Skills

Use this template when creating a new package skill. Replace placeholders with package-specific content.

```yaml
---
name: <package-name>
description: '<Brief description>. WHEN: regenerate <package-name>; modify <package-name>; fix <package-name> bug; add <package-name> feature; <package-name> tsp-client update.'
---
```

The `name` field and directory name MUST match the Python distribution package name (e.g., `azure-search-documents`, `azure-ai-projects`, `azure-mgmt-appconfiguration`).

## Content Principles

- **Keep it static** — no version numbers, no current API versions. Document design and patterns, not release state.
- **Prefer TypeSpec-level customizations over `_patch.py`** — always note when a customization could be expressed by a TypeSpec decorator or emitter option instead.
- **Don't re-document MCP tools** — `azsdk_package_generate_code`, `azsdk_package_build_code`, `azsdk_package_run_check`, `azsdk_package_run_tests`, and `azsdk_customized_code_update` handle generation and related workflows.
- **Focus on the convenience layer** — what does the agent need to know to write/maintain `_patch.py`, hand-written utilities, and async parity correctly.

## Required Sections (in order)

### Common Pitfalls
List 3–5 most dangerous mistakes. This section MUST come first — agents read it before analyzing errors. Always include:
- Never edit files inside `_generated/` or files bearing the `"Code generated by Microsoft (R) Python Code Generator"` header.
- Check the appropriate `_patch.py` FIRST when symbols seem wrong.
- Maintain async parity between sync and `aio/` trees.

### Architecture
Namespace layout, generated vs hand-written distinction (nested `_generated/` vs inline header-identified), non-empty `_patch.py` files and their roles, utility modules. No version numbers.

### After Regeneration
Error categorization table, API version management, breaking-change detection. Do NOT re-document the generation / build / test steps — those are in the MCP tools.

### Post-Regeneration Customizations (if any `_patch.py` is non-empty)
Per-file documentation with "when to update" guidance, dependency on generated symbols, and whether a TypeSpec-level customization is possible instead. Include async counterpart notes.

### Testing Notes
Commands, recorded-test setup (Test Proxy + `assets.json`), environment variables for live tests.

### References
Table linking to `references/*.md` files.
